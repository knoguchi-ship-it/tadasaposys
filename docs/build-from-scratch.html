ゼロから構築ガイド (Build-from-Scratch Guide)

本システムを「何もない状態」から再構築するために、開発者が手元に置くべき情報を 1 本に集約した起点ドキュメント。
2026 年時点のドキュメント・ベストプラクティス（greenfield ワークフロー / arc42＋C4 / ADR / Documentation-as-Code）に基づいて構成。
対象: v1.12.4 GAS + React SPA 作成日: 2026/06/07 読了目安: 30分 / 構築: 半日〜1日

0. このガイドについて

本ガイドは「既に動いているシステムを保守する」ためではなく、同じシステムをゼロから組み立て直すことを想定した 構築起点ドキュメントです。greenfield プロジェクトの標準ワークフロー 要件 (PRD/RD) → アーキテクチャ → 実装 → デプロイ → 検証 に沿って、 「何を・どの順番で・なぜ」用意するかを 1 本にまとめています。

🎯
このガイドのスコープ 既存の詳細ドキュメント（SDD / ADR / RD / Manual など）を置き換えるものではありません。 それらは「完成系の仕様」を記述しており、本ガイドは「ゼロから到達するための順路」と「全体像」を提供します。 各所から既存ドキュメントへリンクし、重複記述（DRY 違反）を避けています。
🔑
本番固有 ID はプレースホルダ表記 本ガイドではスプレッドシート ID・スクリプト ID・デプロイ ID を <SPREADSHEET_ID> のようなプレースホルダで記載します。 現行本番の実値は docs/RUNBOOK.md §6 / docs/HANDOVER.md §7 を参照してください。 Zoom OAuth などの機密情報はコード・ドキュメント・GitHub に置かず、設定シート・PropertiesService で扱います（グランドルール 3）。

対象読者

  • 本システムを別組織・別ドメインへ移植／再構築する開発者
  • 災害復旧（DR）で環境を再作成する運用者
  • アーキテクチャ全体像を短時間で把握したい新規参画者（C4 / ER を参照）

1. 必要ドキュメント体系（ベストプラクティス）

2026 年時点の指針（出典参照）では、開発ドキュメントは 読者（audience）ごとに分け、リリースに紐づけてバージョン管理し、 ADR は不変の意思決定ログとして残すのが定石です。 本プロジェクトは既にこの体系をほぼ満たしています。下表が「ゼロから構築する開発者が手元に揃えるべきドキュメント一式」と、本リポジトリでの対応です。

3. 技術選定の根拠（ADR ダイジェスト）

ゼロから作り直す際、「なぜこの構成なのか」を理解せずに変更すると過去の事故を再発させます。 意思決定は不変ログとして docs/ADR.md に残っています。構築前に必読の判断を抜粋します。

ADR決定ゼロ構築で外してはいけない理由
001認証・認可の二重ガードフロントの表示制御だけでなく バックエンドで権限を強制。UI を迂回した不正操作を防ぐ
002React on GAS（ビルドレス）Babel standalone でブラウザ内トランスパイル。CI/CD ビルド不要、clasp push のみで完結
003タイムスタンプ主キーフォーム回答のタイムスタンプを PK に。手動追加は manual_<epoch> で衝突回避
004React 18.2.0 完全固定19 系混在で Minified React error #31 クラッシュ。全 CDN URL に ?deps=react@18.2.0 必須
007Webapp = USER_ACCESSING / DOMAIN本人メールを確実に取得し、ドメイン外を遮断。過去に誤設定でセキュリティ事故。逸脱厳禁
008下書きはスプレッドシート保存PropertiesService の容量制限を回避。管理者が直接確認可能
009予約送信を廃止時間主導トリガはトリガ作成者として送信され「本人送信」を保証できない。新規構築では実装しない
010管理遷移の統一ゲートキーパーadminTransitionStatus_() に全 STATUS 変更を集約。直接 STATUS 書換は整合性バグの温床
011FullCalendar をグローバルバンドルReact 18 固定と独立させるため UMD 版を <script> 読込（importmap に混ぜない）
012Zoom 時のみ重複検知＋チームカレンダー強制登録Zoom はタダスクとアカウント共有でダブルブッキング事故あり。Zoom 利用時のみ重複ブロック

5. ゼロからの構築手順

greenfield ワークフローに沿い、外側（Google 基盤）から内側（コード）へ、最後にデプロイ・検証の順で構築します。 各 Phase は前 Phase の完了を前提とします。

0
前提アカウント・ツール
必要な権限と開発ツールを揃える
  • Google Workspace ドメイン（本番は tadakayo.jp）— Webapp の access: DOMAIN はこのドメイン内に限定される
  • ドメイン内の 管理者用 Google アカウント（スプレッドシート・GAS の所有者になる）
  • 開発ツール: Node.js 18+、npm、@google/clasp（npm i -g @google/clasp）、git
  • （任意）Zoom 連携を使う場合: Zoom Server-to-Server OAuth アプリの Account ID / Client ID / Client Secret
⚠️
機密情報（Zoom シークレット等）はコード・GitHub に置かない。後続 Phase で設定シート／PropertiesService に格納します。
1
Google 基盤（フォーム → スプレッドシート → IMPORTRANGE）
申込の一次受付とデータストアを用意
  1. 申込フォームを作成（氏名・メール・事業所名・相談内容・都道府県・サービス種別）。回答は専用の「フォーム回答」スプレッドシートに保存される（ID: <FORM_RESPONSES_ID>）
  2. アプリ用スプレッドシートを新規作成（ID: <SPREADSHEET_ID>）。これがアプリの唯一の真実の源
  3. アプリ用シートに 案件リスト シートを作り、A1 に IMPORTRANGE を設定（フォーム回答から必要列のみ左詰め取得）: 
    ={IMPORTRANGE("<FORM_RESPONSES_ID>","フォームの回答 1!A:E"),
  IMPORTRANGE("<FORM_RESPONSES_ID>","フォームの回答 1!H:I")}
    初回はセルの「アクセスを許可」を一度クリックする
  4. Drive フォルダを作成（添付保存先、ID: <ATTACHMENT_FOLDER_ID>）
  5. 必要なら共有カレンダー／チームカレンダーを作成（ID を後で設定シートへ）
2
データモデル（シート作成）
§4 / SDD §1 に従い左詰め・ギャップなしで作る

最低限、起動に必要なのは次の 3 シート。残りは Phase 5 の初期化関数／スキーマ自動移行で生成されます。

  • 案件リスト（Phase 1 で作成済み・書込禁止）
  • サポート記録（19 列。IDX.RECORDS 準拠）
  • タダメンマスタ … 管理者を 1 名登録（B=氏名 / C=メール / D=admin / E=true）
🔁
スキーマ自動移行 初回アクセス時に ensureAttachmentSchema_() が走り、不足列・新シート（年間利用補正、メール下書き等）を SCHEMA_VERSION_='6' まで追補します。手作業で全列を作る必要はありません。

GAS プロジェクト初期化
clasp・Advanced Services・スコープ

# 既存リポジトリ直下に .clasp.json を作成（.gitignore 対象・手動作成）

  "dependencies": { "enabledAdvancedServices": [

  ]},
  "oauthScopes": [
    "https://mail.google.com/",
    "https://www.googleapis.com/auth/calendar",

    "https://www.googleapis.com/auth/spreadsheets",

デプロイ（URL 不変原則）
固定 deploymentId へバージョンアップのみ

初回のみは新規デプロイで Webapp URL と deploymentId を発番します。以降はその固定 ID へ -i 付きで更新し、URL を変えません。

# 0. appsscript.json の webapp が USER_ACCESSING / DOMAIN か確認
# 1. 先に git commit（clasp pull の上書き対策）
git add index.html コード.js appsscript.json
git commit -m "feat: vX.X.X - 説明"

# 2. プッシュ
clasp push --force

# 3-A. 初回だけ：新規デプロイで deploymentId を発番（この値を以後固定）
clasp deploy -d "v1.0.0"   # → <DEPLOYMENT_ID> を控える

# 3-B. 2回目以降：固定 deploymentId へバージョンアップ（URL 不変）
clasp deploy -i <DEPLOYMENT_ID> -d "vX.X.X"

# 4. GAS デプロイ管理画面で「実行=アクセスしているユーザー / アクセス=ドメイン内全員」を目視確認
🔒
運用開始後は新規デプロイ禁止 案内済みのブックマーク・QR・メールリンクが全失効します。必ず -i <固定ID> でバージョンアップしてください（RUNBOOK §1 / ADR-007）。 禁止: USER_DEPLOYING / ANYONE_WITH_GOOGLE_ACCOUNT。

  • Webapp URL にアクセス → ログインできる（タダメンマスタ登録アカウント）
  • 案件一覧（6 タブ）が表示される／フォーム申込が未対応に流れてくる
  • アサイン → 日程確定（Meet 発行）→ 記録 → 完了報告の一連が通る
  • 年間 10 回 / 案件 3 回の上限ガードが効く
  • 管理モードでステータス変更・再アサイン・上限特例が反映される
  • 動作確認後に MAIL_DRY_RUN を false にして実送信を有効化

エンドユーザー観点の確認は docs/Manual.md の操作手順をなぞるのが確実です。

6. セキュリティ要件（5 視点）

セキュアコーディングは次の 5 視点で実装・レビューします（グランドルール 3 / SDD §4）。介護事業所の個人情報を扱うため必須です。

視点本システムでの担保
認証・認可タダメンマスタで本人認証（SEC-01）。管理権限はバックエンドで強制（requireAdmin_）。UI 制御だけに頼らない（ADR-001）
入力検証 / 出力無害化シート書込前に sanitizeForSheet_ で数式インジェクション防止。設定更新はホワイトリストキーのみ（updateSettingsAdmin）
機密情報管理Zoom OAuth 等はコード・GitHub に置かず設定シート / PropertiesService で保持。ハードコーディング原則禁止
データ保護 / 最小権限Webapp は DOMAIN 限定。案件リストは書込禁止で IMPORTRANGE を保護。担当・サブ担当・管理者以外の編集を ensureCaseEditableByActor_ で拒否
監査性 / エラー処理管理操作は監査ログへ BEFORE/AFTER_JSON 付きで記録。失敗時は安全側に停止
