# アーキテクチャ意思決定記録 (ADR)

本プロジェクトにおける重要な技術選定および設計判断の記録。
各 ADR は Y-Statements 形式で記録する: 「**文脈**において、**懸念事項**に直面したため、**決定事項**を採用した。これにより**長所**が得られるが、**短所**を許容する。」

---

## ADR-001: 認証・認可の二重ガード (Security)

**ステータス:** 承認済み

**文脈:** GAS Webapp は `executeAs: USER_ACCESSING` で動作し、ブラウザ側の UI と GAS バックエンドの両方で権限チェックが必要。

**決定事項:** フロントエンドの表示制御（UI の非表示・無効化）とバックエンドの権限チェック（`requireAdmin_()`・`ensureCaseEditableByActor_()`）を両方実装する二重ガードを採用する。

**長所:** フロントのバイパスによる不正操作をバックエンドで完全に防御できる。
**短所:** 同じロジックを2箇所に保守する必要がある。

## ADR-002: React on GAS (Frontend Architecture)

**ステータス:** 承認済み

**文脈:** GAS の `HtmlService` では Node.js ビルドパイプラインが使えない。しかし React の SPA を実現したい。

**決定事項:** Babel standalone を CDN から読み込み、ブラウザ内で JSX をトランスパイルする。ビルドステップなしで React を動作させる。

**長所:** clasp によるデプロイのみで完結。CI/CD パイプライン不要。
**短所:** ブラウザ内トランスパイルによる初回読み込みの遅延。バンドルサイズ最適化不可。

## ADR-003: タイムスタンプによる主キー管理 (Data Modeling)

**ステータス:** 承認済み

**文脈:** Google フォームの回答タイムスタンプを一意識別子として使用する必要がある。

**決定事項:** フォームの送信タイムスタンプを案件の PK として使用する。手動追加案件は `manual_` + エポックミリ秒 で衝突を回避する。

**長所:** フォームの IMPORTRANGE 構造を変更せずに PKを取得できる。
**短所:** タイムスタンプの衝突リスク（実運用上は無視できる程度）。

## ADR-004: ライブラリバージョンの固定 (Dependency Management)

**ステータス:** 承認済み

**文脈:** `esm.sh` 等のCDNでバージョン指定が緩いと、依存解決の過程で React 19 等が読み込まれる。React のインスタンスが複数存在すると `Minified React error #31` が発生してアプリがクラッシュした。

**決定事項:** 全外部ライブラリのURLにバージョンを完全固定し、`?deps=react@18.2.0` で依存も明示する。

```
良い例: https://esm.sh/lucide-react@0.330.0?deps=react@18.2.0
悪い例: https://esm.sh/lucide-react
```

**長所:** 予期せぬ破壊的変更によるクラッシュを完全防止。
**短所:** ライブラリ更新時は手動でURL書き換えが必要。

## ADR-005: インライン編集UIパターン（Popover + Overlay）

**ステータス:** 承認済み — v1.9.0

**文脈:** 管理者がステータス・担当者・上限値を変更する際、ページ下部のフォームへスクロールする操作が煩雑だった。

**決定事項:** `fixed inset-0 z-40` の透明オーバーレイ + 相対配置のポップオーバーパターンを採用する。

- バッジをクリック → `inlineEditField` state でポップオーバーを制御
- オーバーレイクリックで閉じる
- `useEffect` + `keydown` リスナーで Escape キーによるクローズ
- 案件切替時は `useEffect([selectedCaseId])` で自動クリア

**長所:** モーダルより軽量。スクロール不要で直感的。
**短所:** `inlineEditField: null | 'status' | 'staff' | 'caseLimit' | 'annualLimit'` の state 管理が増加。

## ADR-006: 新着バッジの実装方式（ID差分方式 vs タイムスタンプ方式）

**ステータス:** 承認済み — v1.9.0

**文脈:** タブに「最後に閲覧した時点から新しい案件が追加・移動した場合」にバッジ表示する機能を実装するにあたり方式を検討した。

**選択肢:**
- **A. タイムスタンプ比較:** `case.timestamp > lastViewed` でカウント → ステータス変更による案件移動を検出不可
- **B. ケースIDセット差分:** そのタブに存在したIDセットを保存し現在との差分をカウント → ステータス変更も検出可能

**決定事項:** B. ケースIDセット差分方式を採用する。

**長所:** ステータス変更による案件移動も正確に検出できる。
**短所:** 初回ロード時（localStorage未設定）は全ケースが新着として表示される（期待された動作）。

## ADR-007: Webapp 設定 — USER_ACCESSING / DOMAIN の採用

**ステータス:** 承認済み — v1.11.x 以前（確認: 2026/05/10）

**文脈:** GAS Webapp の `executeAs` には `USER_DEPLOYING`（デプロイ者の権限で実行）と `USER_ACCESSING`（アクセス者の権限で実行）の2択がある。`access` には `ANYONE`（誰でもアクセス可）と `DOMAIN`（ドメイン限定）がある。

**決定事項:** `executeAs: USER_ACCESSING` / `access: DOMAIN` を採用する。

**長所:**
- `USER_ACCESSING` により `Session.getActiveUser()` が確実に実際のログインユーザーのメールを返す。認証の信頼性が高い。
- `DOMAIN` により tadakayo.jp ドメイン外からのアクセスを完全遮断できる。

**短所:**
- `access: DOMAIN` のため外部ユーザー（ドメイン外の tadakayo.jp 以外）はアクセス不可（このシステムでは意図した制限）。
- 時間主導トリガはトリガ作成者の権限で動作するため、Webアプリのアクセスユーザー本人として Gmail 送信する用途には使えない。この制約により予約送信は v1.12.1 で廃止した。

**注意:** 旧 HANDOVER.md には誤って `USER_DEPLOYING` / `ANYONE` と記載されていたが、実際の appsscript.json は `USER_ACCESSING` / `DOMAIN` であることを clasp pull で確認済み（2026/05/10）。

## ADR-008: メール下書きのスプレッドシート実装

**ステータス:** 承認済み — v1.11.0

**文脈:** メール送信前の下書きを保存する仕組みが必要になった。選択肢として「PropertiesService（GASのユーザー設定ストア）」と「スプレッドシートの専用シート」を検討した。

**選択肢:**
- **A. PropertiesService:** GAS の UserProperties を使用。1ユーザーあたり最大 50KB の制限あり。本文が長いと溢れる可能性。
- **B. スプレッドシート専用シート:** 「メール下書き」シートを追加。容量制限なし。管理者が直接確認・クリーンアップ可能。

**決定事項:** B. スプレッドシート専用シートを採用する。複合キー `CASE_ID + STAFF_EMAIL + MODE + THREAD_ID` で上書き保存し、1セッション1下書きを維持する。

**長所:** 容量制限なし。管理者がスプレッドシートで直接確認・クリーンアップ可能。既存の DB パターンと一貫している。
**短所:** スプレッドシートへの読み書きコストが発生する（他のシート操作と比較して誤差範囲）。

## ADR-009: 予約送信のキューアーキテクチャ（スプレッドシートキュー + 時間主導トリガ）

**ステータス:** 廃止 — v1.12.1

**文脈:** 指定日時にメールを自動送信する機能が必要になった。GAS の環境でどのようにキュー処理を実装するか検討した。

**選択肢:**
- **A. Cloud Tasks / Pub/Sub:** GCP のマネージドキューサービス。信頼性が高い。追加の GCP 設定・認証・費用が必要。
- **B. スプレッドシートキュー + 時間主導トリガ:** 予約をスプレッドシートに保存し、5分間隔のトリガで処理する。GAS のみで完結。
- **C. GAS の特定時刻トリガ:** 1件ごとにトリガを作成。GAS のトリガ上限（20個/プロジェクト）に抵触するリスク。

**決定事項:** B. スプレッドシートキュー + 5分間隔の時間主導トリガを採用する。`LockService.getScriptLock()` で並行実行を防止。`sending` 状態で 10 分以上経過した行はスタックとみなして再処理する。

**長所:** 追加の外部サービス不要。GAS のみで完結。スプレッドシートで予約状況を可視化・管理できる。
**短所:** 送信タイミングに最大5分の遅延が生じる。GAS の実行時間上限（6分）内で処理するため1回のトリガで最大 `SCHEDULED_BATCH_LIMIT=20` 件に制限。

**廃止理由（v1.12.1）:** 時間主導トリガはトリガ作成者のアカウントで実行される。`Gmail.Users.Messages.send(..., 'me')` の `me` が予約者本人ではなくトリガ作成者になるため、「Webアプリにアクセスしている本人のメールアドレスから送信する」要件を満たせない。予約送信 UI/API は撤去し、旧キューは `disablePendingScheduledEmails()` で `disabled` 化する。

## ADR-010: 管理機能ステータス遷移の統一ゲートキーパーパターン

**ステータス:** 承認済み — v1.11.6

**文脈:** v1.11.5 以前の管理機能（`setCaseStatusAdmin` / `reassignCaseAdmin` / `updateCaseDataAdmin`）は STATUS 列のみを直接書き換えていた。このため、`completed → inProgress` でHISTORY が保存されない、`→ unhandled` でスタッフ情報が残存するなど、11件の致命的・重大なデータ整合性バグが存在した。

**選択肢:**
- **A. 各関数に個別にロジックを追加**: 関数ごとに遷移ロジックを実装。重複コードが増加し、将来の漏れリスクが高い。
- **B. 統一ゲートキーパー関数**: `adminTransitionStatus_()` を1つ新設し、全ての STATUS 変更をここに集約。各関数はこれに委譲する。

**決定事項:** **B. 統一ゲートキーパーパターン**を採用する。

`adminTransitionStatus_(recordSheet, rowIndex, fromStatus, toStatus, actor, options)` が `fromStatus × toStatus` の全組み合わせを処理し、遷移に応じた DB 操作セット（HISTORY 保存・フィールドクリア・上限自動調整など）を確実に実行する。

**長所:**
- 遷移ロジックが1箇所に集約され、将来の修正・追加が容易。
- 全ての管理者経由 STATUS 変更が確実に整合性チェックを受ける。
- 関数の戻り値で変更後の全フィールドを返すため、フロントエンドの楽観的更新も正確になる。
- `completed → inProgress` で上限超過時に `caseLimitOverride` を自動調整するロジックも一元管理。

**短所:** 関数内の分岐が多く、将来の新ステータス追加時に分岐の追加が必要。

## ADR-011: FullCalendar 6.1.x をグローバルバンドルで埋込み

**ステータス:** 承認済み — v1.11.9

**文脈:** v1.11.7 までの日程確定モーダルは `<input type="datetime-local">` のみで、空き状況を視覚的に確認できなかった。タダスクや既存案件との重複を毎回手動で確認する運用負担があり、日程被りの事故リスクがあった。

**選択肢:**
- **A. importmap 経由で `@fullcalendar/core` 等を ES Module として読み込む**: 既存の React 用 importmap と同じパターン。ただし `?deps=react@18.2.0` 強制と独立した依存ツリーになりバージョン管理が複雑化。
- **B. `@fullcalendar/react` ラッパーを使う**: React ネイティブで宣言的だが、React 18.2.0 厳格固定（ADR-004）と相性が悪く、CDN 解決の工夫が必要。
- **C. グローバル UMD バンドル（`fullcalendar@6.1.18/index.global.min.js`）を `<script>` で読み込み**: core/daygrid/timegrid/interaction が同梱。既存の React コードと完全独立。

**決定事項:** **C. グローバルバンドル**を採用する。

`window.FullCalendar.Calendar` を `useRef` + `useEffect` でマウントするラッパーコンポーネント `ScheduleAvailabilityCalendar` を実装。

**長所:**
- React 18.2.0 固定制約と完全に独立して動作
- CDN キャッシュ後はバンドル取得コストが消滅
- 公式リファレンス実装と同じパターンで学習コスト最小
- バンドル内のプラグインバージョンが整合済みで、組合せ起因のバグ無し

**短所:**
- 初回読込みで ~200KB 増加（jsDelivr CDN 配信）
- 宣言的でない（命令的 imperative API を React の useEffect 内で扱う）

## ADR-012: 方法=Zoom 時のチームカレンダー強制登録 + 重複検知

**ステータス:** 承認済み — v1.11.8 / v1.11.9-r2 で確定

**文脈:** Zoom はタダスク業務とアカウントを共有しており、過去にダブルブッキングで会議が衝突した事故があった。一方、GoogleMeet / 電話 / 対面 / メール等は会議リソース競合の概念がなく、重複しても運用上問題ない。

**選択肢:**
- **A. すべての方法で重複チェック**: 一貫性は高いが、電話・対面で誤検知が出てユーザー摩擦が大きい。
- **B. Zoom + GoogleMeet+カレンダー登録 で重複チェック**: Phase 2 当初案。Meet も会議室扱いだが、複数 Meet 同時開催可能なので過剰。
- **C. 方法=Zoom 時のみ重複チェック**: Zoom リソース競合の防止に焦点化。他の方法はチェックなし。

**決定事項:** **C. Zoom 限定** を採用（v1.11.9-r2 で確定）。

加えて、Zoom 利用時は **`TEAM_CALENDAR_ID` への登録を必須化**する（`useCalendar` チェックの有無に関わらず）。これにより `getScheduleAvailability` がチームカレンダーを真実のソースとして重複検知できる。

**長所:**
- Zoom 衝突リスクが最重要なので、そこに集中投資
- 他の方法はユーザー操作を妨げない
- 個人カレンダー登録は引き続き選択制で柔軟性維持
- 監視対象は `TEAM_CALENDAR_ID` + `DISPLAY_CALENDARS_JSON` で運用可変

**短所:**
- 仕様変更：Zoom + useCalendar=OFF が「何もしない」→「チームのみ登録」に挙動変化（Manual.md / CHANGELOG で周知済み）
- `TEAM_CALENDAR_ID` にスタッフが書込権限を持つ前提（既存運用で満たされている）
