## 1. システム概要

### 1.1 何のためのシステムか

タダサポ管理システムは、**介護事業所向けの無料 IT サポート**を運営する NPO（タダカヨ）の業務を支えるための管理システムです。介護現場の職員から「パソコンやアプリの使い方が分からない」といった相談が Google フォームで届き、タダメン（ボランティアの運営スタッフ）が担当を引き受け、日程を調整し、オンライン等で支援し、その記録を残します。本システムはこの一連の流れを一元管理します。

技術的には、**Google Apps Script（GAS）上で動作する React 製の Web アプリ**であり、**Google スプレッドシートを唯一のデータベース（信頼できる唯一の情報源）**として使います。専用のサーバーやデータベース製品を持たず、Google Workspace の中だけで完結するのが大きな特徴です。

### 1.2 全体像（構成要素と外部連携）

システムは大きく次の3つの実体で構成されます。

- **フロントエンド（`index.html`）** — ブラウザで動く React の画面。1ファイルに収まっており、開発用のモックデータも内蔵しているため、GAS なしでもローカルで画面を確認できます。
- **バックエンド（`コード.js`）** — GAS 上で動くサーバー側のロジック。Web アプリの入口 `doGet()` と、画面から呼び出される多数の公開関数からなります。
- **データストア（Google スプレッドシート）** — 設定・案件・記録・スタッフ・メール履歴・監査ログなどを保持するシート群。

外部サービスとの連携は次のとおりです。

| 連携先 | 役割 |
|--------|------|
| Google フォーム ＋ 回答スプレッドシート | 依頼の一次受付。回答が IMPORTRANGE 経由で案件リストに流れ込む |
| Gmail | タダメン本人のアカウントからの送信、スレッド管理 |
| Google カレンダー / Meet | 支援日程の登録、Meet 会議 URL の発行 |
| Zoom（任意） | Server-to-Server OAuth で Zoom 会議を作成（使う場合のみ） |
| Google ドライブ | 支援記録に添付するファイルの保存 |

### 1.3 利用者と表示モード

利用者はブラウザでアクセスし、Google アカウント（運営の所属ドメイン）で認証されます。タダメンには役割（admin / staff）があり、画面には3つの表示モードがあります。

- **通常モード** — 自分が担当している案件のみを表示・操作（未対応案件は全員で共有）。
- **閲覧モード** — 全案件を表示するが、書き込み操作はできない（読み取り専用）。
- **管理モード** — 管理者のみ。全案件を表示し、ステータス変更・再アサイン・設定変更など管理専用の操作ができる。

## 2. 業務フロー（何を、どう回すか）

本システムが支える業務は、受付から完了までを次の4フェーズで進みます。実際の画面・機能はこのフローに対応しています。

### フェーズ1：受付
1. 依頼者（介護事業所の職員）が Google フォームに相談内容を入力する。
2. フォーム回答が回答スプレッドシートに自動追記され、IMPORTRANGE で「案件リスト」シートに取り込まれる。
3. アプリは案件リストを読み込み、「未対応」案件として一覧に表示する。

### フェーズ2：アサインと日程調整
1. タダメンは未対応案件の中から、自分が担当する案件を選ぶ（セルフアサイン）。
2. このとき**回数制限のチェック**が入る。年間の利用回数が上限未満なら「初回日程調整メール」へ、上限に達していれば「上限お断りメール」へ進む。
3. アプリ上のテンプレートを使い、担当者本人の Gmail から初回連絡メールを送る。送信内容はメール履歴に記録される。
4. 依頼者との具体的な日時のやり取りは、アプリの外（通常の Gmail）で行う。
5. 日時が決まったらアプリに入力する。保存時に担当者カレンダーと共有カレンダーへ予定を作成し、Meet URL を発行する。
6. 日時と Meet URL を載せた確定メールをアプリから送る（履歴に記録）。

### フェーズ3：実施と記録
1. 発行された Meet URL 等で実際の支援を行う。
2. 支援後、アプリで実施記録（内容・対応方法・備考・添付ファイルなど）を入力・保存する。
3. 完了報告メールをアプリから送る（履歴に記録）。

### フェーズ4：継続・完了・お断り
- 完了した案件は、必要に応じて「再開」して継続支援できる（1案件あたり上限回数まで）。
- 年間の利用回数が上限に達した場合は、丁寧なお断りメールを送り、案件を「対応不可（rejected）」にする。
- 途中・完了後に取りやめる場合は「キャンセル（cancelled）」にする。

## 3. ビジネスルール（守るべき業務上の決まり）

業務を正しく回すために、システムは次のルールを強制します。これらは設定値で調整できるものと、固定のものがあります。

### 3.1 回数制限

- **1案件あたりの対応回数**：標準で最大 **3 回**。再開を繰り返してもこの回数を超えられない。
- **年間の利用回数**：同一メールアドレスにつき、1 年度で最大 **10 回**。

「年間」は **4 月 1 日始まりの年度**で数えます。年間利用回数は、**同じメールアドレス（大文字小文字・前後空白を無視して正規化）かつ同じ年度**の、「対応中（inProgress）」「完了（completed）」案件の対応回数（supportCount）を合算した値です。Google フォームからの申込と、管理者が手動追加した案件も、同一メール＋同一年度であれば合算されます。

### 3.2 上限値の優先順位

上限は「案件ごとの特例 → 全体設定 → ハードコードの既定値」の順で決まります。

```
1案件の対応回数上限 ＝ 案件特例(caseLimitOverride) ＞ 全体設定 ＞ 既定 3
年間の利用回数上限 ＝ 案件特例(annualLimitOverride) ＞ 全体設定 ＞ 既定 10
```

つまり、特定の案件だけ上限を緩めたいときは、その案件に特例値を設定できます（管理者操作）。特例を空にすると全体設定に戻ります。

### 3.3 ステータスと遷移

案件のステータスは次の5種類です。

| ステータス | 意味 |
|-----------|------|
| `unhandled` | 未対応（まだ誰も担当していない） |
| `inProgress` | 対応中 |
| `completed` | 完了 |
| `rejected` | 対応不可（回数超過でお断り済み） |
| `cancelled` | キャンセル |

遷移は次のとおりです。

```
unhandled → inProgress → completed → （再開すると inProgress に戻る。上限回数まで）
unhandled → rejected（回数超過のお断りメール送信後）
inProgress / completed → cancelled（担当者・管理者がキャンセル）
```

### 3.4 回数超過時の挙動

- 未対応で、かつ年間利用が上限以上の案件は、「担当する」ボタンを隠し、「回数超過」ボタンのみを表示する。
- 完了済みで、かつ年間利用が上限以上の案件は、「再開」ボタンを表示しない。

## 4. アーキテクチャと技術選定の理由

ここでは「なぜこの構成なのか」を説明します。これらは過去の試行錯誤や事故を踏まえた確定済みの判断であり、**理由を理解せずに変更すると過去の不具合を再発させます**。

### 4.1 技術スタック

| 領域 | 採用技術 |
|------|---------|
| フロントエンド | React 18.2.0 ＋ Babel standalone（CDN, importmap で読込） |
| アイコン | lucide-react 0.330.0（`?deps=react@18.2.0` 必須） |
| カレンダーUI | FullCalendar 6.1.18（グローバル UMD バンドルを `<script>` で読込） |
| CSS | Tailwind CSS（CDN） |
| バックエンド | Google Apps Script（V8 ランタイム） |
| データストア | Google スプレッドシート |
| メール | Gmail Advanced Service（API v1） |
| カレンダー | Calendar Advanced Service（v3）＋ CalendarApp |
| 会議 URL | Google Meet（Calendar API）／ Zoom API |
| デプロイ | clasp |
| テスト | Playwright（E2E）＋ Jest（単体） |

### 4.2 主要な設計判断（その理由）

**(1) GAS の上で React を動かし、ビルド工程を持たない。**
GAS の `HtmlService` では Node.js のビルドパイプラインが使えません。そこで Babel standalone を CDN から読み込み、**ブラウザの中で JSX を変換**します。これにより、CI/CD のビルドが不要になり、`clasp push`（コードのアップロード）だけでデプロイが完結します。代償として初回読み込みがやや遅く、バンドル最適化はできません。

**(2) React を 18.2.0 に完全固定する。**
`esm.sh` などの CDN はバージョン指定が緩いと依存解決の過程で React 19 を引き込むことがあります。React のインスタンスが複数できると `Minified React error #31` でアプリが**クラッシュ**します。これを防ぐため、すべての CDN URL にバージョンを完全固定し、`?deps=react@18.2.0` で依存も明示します。良い例：`https://esm.sh/lucide-react@0.330.0?deps=react@18.2.0`。

**(3) 認証・認可は「フロントとバックエンドの二重ガード」にする。**
画面側でボタンを隠すだけでは、画面を迂回した不正操作を防げません。そこで**バックエンドでも必ず権限を確認**します（詳細は第9章）。同じ判定を2か所に持つ保守コストはありますが、安全性を優先します。

**(4) Web App の実行・アクセス設定を固定する。**
Web App は「アクセスしているユーザーとして実行（USER_ACCESSING）」かつ「所属ドメイン内に限定（DOMAIN）」に固定します。これにより本人のメールアドレスを確実に取得でき、ドメイン外からのアクセスを遮断できます。**過去にこの設定を誤って運用しセキュリティ事故が起きた**ため、逸脱は固く禁じます（第9章で詳述）。

**(5) 案件の主キーはフォーム回答のタイムスタンプ。**
フォームの送信タイムスタンプを案件の一意キー（PK）として使います。手動追加案件は `manual_<エポックミリ秒>` という形で衝突を避けます。これにより IMPORTRANGE の構造を変えずに PK を得られます。

**(6) メール下書きはスプレッドシートに保存する。**
GAS のユーザー設定ストア（PropertiesService）は容量制限（1ユーザー約50KB）があり、長い本文で溢れる恐れがあります。そこで専用シートに保存し、容量制限を回避しつつ管理者が直接確認・整理できるようにしています。

**(7) メールの予約送信機能は廃止した。**
時間主導トリガーは「トリガー作成者のアカウント」で実行されるため、「アクセスしている本人から送信する」という要件を満たせません。本人送信を保証できないため予約送信は廃止しました。**新規に再実装してはいけません**。即時送信と下書き保存で運用します。

**(8) 管理者によるステータス変更は1つの関数に集約する。**
以前は管理機能がステータス列だけを直接書き換えていたため、「完了→対応中で履歴が残らない」「未対応に戻すとスタッフ情報が残る」などの整合性バグが多発しました。現在は `adminTransitionStatus_()` という統一ゲートキーパー関数に全てのステータス変更を集約し、遷移ごとに必要な処理（履歴保存・項目クリア・上限自動調整など）を確実に行います。

**(9) FullCalendar はグローバルバンドルで読み込む。**
React 18 固定（上記(2)）と独立させるため、FullCalendar は ES Module の importmap に混ぜず、UMD のグローバルバンドルを `<script>` で読み込みます。これでバージョン競合を避けられます。

**(10) 日程の重複検知は Zoom のときだけ行う。**
Zoom は別業務とアカウントを共有しており、過去にダブルブッキング事故がありました。一方 Google Meet・電話・対面・メールは会議リソースの競合概念がありません。そこで**方法が Zoom のときのみ重複を検知してブロック**し、Zoom 利用時はチームカレンダーへの登録を必須化して、重複判定の基準を一元化しています。

## 5. データモデル（スプレッドシートの全構造）

データは1つの Google スプレッドシートに保持します。すべてのシートは**列を左詰め・隙間なし**で並べ、列番号はコード内の `IDX` 定数を唯一の基準とします。以下に全シートの完全な列定義を示します。

### 5.1 シート一覧

| シート名 | 役割 |
|---------|------|
| 設定 | すべての運用設定値（Key-Value 形式） |
| 案件リスト | フォーム回答の取り込み（IMPORTRANGE、**書き込み禁止**） |
| 案件補正 | 管理者による案件情報の手動補正 |
| 案件手動追加 | アプリから手動追加した案件 |
| サポート記録 | 対応ステータス・担当・記録のメインシート（19列） |
| タダメンマスタ | スタッフ一覧（認証・権限） |
| メール履歴 | 送信メールのバックアップ |
| 監査ログ | 管理者操作の証跡 |
| メール下書き | 送信前メールの一時保存（11列） |
| 予約送信キュー | 予約送信は廃止。履歴保持のみ（16列） |
| 年間利用補正 | 年度利用回数の手動補正（5列） |

### 5.2 設定（Key-Value ストア）

システム全体の運用設定を保持します。**A列のキーは変更不可、C列の値だけを編集**します。主な構造は A=設定キー / B=項目名 / C=設定値 / D=入力例 / E=説明。代表的なキーの一覧は付録13.1にまとめています。

メールテンプレートでは次のタグが本文中で展開されます：`{{名前}}` `{{事業所名}}` `{{担当者名}}` `{{相談内容}}` `{{日程}}` `{{対応方法}}` `{{URL}}`。

### 5.3 案件リスト（IMPORTRANGE・書き込み禁止）

依頼の一次データ。Google フォーム回答から IMPORTRANGE で取り込みます。**このシートに `setValue` 等で書き込むと IMPORTRANGE の数式が壊れるため、絶対に書き込んではいけません。** 管理者による補正は「案件補正」シートで行います。

| 列 | 論理名 | 内容 |
|----|--------|------|
| A | PK | 主キー（フォーム送信タイムスタンプ） |
| B | EMAIL | 依頼者メール |
| C | OFFICE | 介護事業所名 |
| D | NAME | 依頼者氏名 |
| E | DETAILS | 相談内容 |
| F | PREFECTURE | 都道府県 |
| G | SERVICE | サービス種別 |

### 5.4 サポート記録（メイン・全19列）

タダサポ側の対応状況を管理する中心シート。案件リストと 1:0〜1 で対応します。

| 列 | 論理名 | 内容 |
|----|--------|------|
| A | FK | 外部キー（案件リスト.PK） |
| B | STATUS | unhandled / inProgress / completed / rejected / cancelled |
| C | STAFF_EMAIL | 担当者メール |
| D | STAFF_NAME | 担当者名 |
| E | DATE | 実施予定日時 |
| F | COUNT | 対応回数（1〜上限） |
| G | METHOD | 対応方法（GoogleMeet / Zoom / 電話 / 対面 / メール 等） |
| H | BUSINESS | 事業種別 |
| I | CONTENT | 実施内容 |
| J | REMARKS | 備考 |
| K | HISTORY | 過去回の記録（JSON） |
| L | EVENT_ID | カレンダーイベントID |
| M | MEET_URL | Meet / Zoom URL |
| N | THREAD_ID | Gmail スレッドID（複数はカンマ区切り） |
| O | ATTACHMENTS | 現在回の添付ファイル（JSON、最大5件） |
| P | CASE_LIMIT_OVERRIDE | 案件ごとの対応回数上限特例（null=全体設定） |
| Q | ANNUAL_LIMIT_OVERRIDE | 案件ごとの年間上限特例（null=全体設定） |
| R | TOOLS | 対応ツール（JSON） |
| S | SUB_STAFF | サブ担当（OJT）（JSON、最大1名） |

HISTORY 列（過去回の記録）は、1回分が次の形の JSON エントリです：`round`（回数）, `scheduledDateTime`, `method`, `content`, `remarks`, `meetUrl`, `attachments`, `tools`, `staffName`, `staffEmail`。

### 5.5 タダメンマスタ（認証・権限）

スタッフの一覧であり、**ログイン認証と権限判定の基礎**です。

| 列 | 論理名 | 内容 |
|----|--------|------|
| A | STAFF_ID | 一意識別子 |
| B | NAME | 表示名 |
| C | EMAIL | 認証キー（このメールでログイン照合） |
| D | ROLE | admin / staff |
| E | IS_ACTIVE | true / false（無効化フラグ） |

### 5.6 案件補正 / 案件手動追加

案件リストは書き込み禁止のため、管理者の補正値は「案件補正」シートに別途持ち、表示時にマージします（補正に値があればそれを優先）。空欄のフィールドは「補正なし＝元の値を使う」を意味します。列構成は案件リストと同じ（A=PK、B〜G=EMAIL〜SERVICE）。「案件手動追加」はアプリから足した案件で、PK は `manual_<エポックミリ秒>`。

### 5.7 メール履歴 / 監査ログ / メール下書き / 年間利用補正

**メール履歴**（送信メールのバックアップ）：A=CASE_ID, B=SEND_DATE, C=SENDER_EMAIL, D=SENDER_NAME, E=RECIPIENT_EMAIL, F=SUBJECT, G=BODY。

**監査ログ**（管理操作の証跡）：A=TIMESTAMP, B=ACTOR_EMAIL, C=ACTOR_NAME, D=ACTION, E=TARGET_TYPE, F=TARGET_ID, G=BEFORE_JSON, H=AFTER_JSON。**管理者の操作は変更前後の値とともにここへ記録**されます。

**メール下書き**（11列）：A=DRAFT_ID, B=CASE_ID, C=STAFF_EMAIL, D=MODE, E=THREAD_ID, F=SUBJECT, G=BODY, H=CC, I=BCC, J=TOOLS, K=UPDATED_AT。`CASE_ID＋STAFF_EMAIL＋MODE＋THREAD_ID` の複合キーで上書き保存します。

**年間利用補正**（5列、v1.12.4）：A=EMAIL, B=FISCAL_YEAR, C=ADJUSTMENT（補正量）, D=UPDATED_BY, E=UPDATED_AT。年間利用回数は派生値（自動計算）のため、管理者が実数を直したいときは「目的値 − 自動計算値 ＝ 補正量」を保存し、表示は「自動計算値＋補正量」とします。実効値は0未満になりません。

**予約送信キュー**（16列）：予約送信機能の廃止に伴い新規登録は行わず、履歴と無効化確認のためだけに残しています。

## 6. 機能仕様（バックエンド関数とUI）

画面（`index.html`）は `google.script.run` を通じてバックエンド（`コード.js`）の公開関数を呼びます。ここでは主要な機能をグループごとに説明します。

### 6.1 起動・データ取得

- **`doGet()`** — Web App の入口。HTML を返す際に初期データを `window.__INITIAL_DATA__` として埋め込み、初回通信を1往復減らして高速化します。
- **`getInitialData()`** — 起動時にユーザー情報・案件・マスタ・下書き有無・強制CCをまとめて返す。このとき**スキーマの自動移行**（不足列や新シートの追補）も行います。
- **`getAllCasesJoined()`** — 案件リスト＋手動追加＋補正＋サポート記録＋メール履歴を結合し、削除済みを除外して返す。各案件に「今年度の累計利用回数」を付けます。
- **`getMasters()`** — 対応方法・事業種別・都道府県・スタッフ一覧・上限値・対応ツール・メールテンプレート等のマスタを返す。
- **`getStaffByEmail(email)`** — メールからスタッフ情報を引く。

### 6.2 案件の操作

案件の更新系は原則として**担当者・サブ担当・管理者のみ**が実行できます（バックエンドの `ensureCaseEditableByActor_()` で強制。一部はメイン担当/管理者などの個別条件）。

- **`assignCase(caseId, user, tools)`** — メールなしで担当を引き受ける（記録がなければ inProgress / 回数1 で新規作成）。
- **`assignAndSendEmail(...)`** — 担当を引き受け、初回メールも送る。Gmail でスレッドを開始し、スレッドIDを保存。
- **`updateSupportRecord(recordData)`** — 記録の更新。カレンダー連携・ステータス変更・添付更新（最大5件）を含む。
- **`reopenCase(caseId, user)`** — 完了案件の再開。現在回を HISTORY に退避し、回数+1、対応中に戻し、各項目をクリア。上限到達時はエラー。
- **`rollbackCurrentRound(caseId)`** — 2回目以降の対応回を取り消し、前回完了状態へ戻す（1回目は不可）。
- **`cancelCase(caseId)`** — 案件をキャンセル。未アサイン案件は管理者のみ。
- **`declineCase(...)`** — 回数超過のお断りメールを送り、ステータスを rejected にする。
- **`updateSupportHistory(caseId, roundIndex, patch)`** — 過去回（HISTORY）の記録を編集（担当者・サブ担当・管理者のみ。`ensureCaseEditableByActor_` で権限確認）。
- **`updateSubStaff(caseId, subStaffArray)`** — サブ担当（OJT、最大1名）を設定（**メイン担当または管理者のみ**）。
- **`addManualCase(payload)`** — 案件を手動追加（**管理者のみ**。PK は `manual_…`）。

### 6.3 メール

- **`sendNewCaseEmail(...)`** — 新規スレッドでメール送信。
- **`sendCaseEmail(...)`** — 既存スレッドへ返信（未指定なら新規）。
- **`getThreadMessages(caseId)`** — 案件の全 Gmail スレッドのメッセージを取得（スレッドIDがなければメール履歴から代替表示）。
- **`saveDraft / loadDraft / deleteDraft / listDraftsForCase`** — メール下書きの保存・読込・削除・一覧。送信成功時に自動削除。

メール送信時、差出人（From）はタダメンマスタの氏名を使い、文字化け防止のため RFC 2047 でエンコードします。

### 6.4 日程・会議

- **`getScheduleAvailability(rangeStart, rangeEnd)`** — 指定期間の空き状況を返す（カレンダー表示用）。
- **`checkScheduleConflict(start, dur, excludeId)`** — 重複（バッファ込み）を判定。
- **`createGoogleMeetEvent(...)`** — Meet 付きカレンダーイベントを作成し、説明欄にアプリ URL を追記。
- **`createZoomMeeting(...)`** — Zoom の Server-to-Server OAuth で会議を作成。Zoom 失敗時もカレンダー作成は続行（分離設計）。
- **`updateMeetUrl(caseId, newUrl)`** — URL を変更し、カレンダーの説明欄も同期。

### 6.5 管理者機能（権限が必要）

これらは**バックエンドで管理者権限を確認**（`requireAdmin_()`）してから実行されます。更新を伴う管理操作は監査ログに記録されます（`getAdminPanelData` は参照のみ）。

- **`getAdminPanelData()`** — 管理パネルの初期データ（スタッフ一覧・編集可能設定・監査ログ）。参照のみ。
- **`upsertStaffMember(payload)`** — 既存スタッフの権限更新（新規追加は不可。マスタは外部管理）。
- **`deactivateStaffMember(email)`** — スタッフを論理無効化。
- **`updateSettingsAdmin(patch)`** — **許可されたキーのみ**設定を更新（ホワイトリスト外は拒否）。
- **`reassignCaseAdmin(caseId, staffEmail)`** — 再アサイン（有効なスタッフのみ）。
- **`setCaseStatusAdmin(caseId, status)`** — 任意のステータスへ変更（内部で統一ゲートキーパーを経由）。
- **`updateCaseDataAdmin(caseId, payload)`** — 案件データの直接編集（変更されたキーのみ反映。上限特例に null を入れると全体設定に戻る）。
- **`setAnnualUsageCountAdmin(caseId, desiredCount)`** — 年度利用回数の実数を手動修正（補正シートへ反映）。
- **`deleteCaseAdmin(caseId)`** — 案件をソフトデリート（削除IDリストに追記）。

### 6.6 UI のポイント

- **案件一覧（6タブ）** — 未対応 / 対応中 / 完了 / キャンセル / 対応不可 / 全て。各タブには「前回見たときからの新着」を示すバッジ（+N）が出ます。新着は、前回保存した案件IDの集合と現在の集合の差分で計算します（ステータス変更による移動も検出できる方式）。新着の既読状態はブラウザの localStorage に保存するため、ブラウザをまたぐと同期されません。
- **検索・フィルタ** — キーワード検索は常時表示。適用中の条件はチップ（pill）で一覧表示・個別削除でき、詳細条件（期間プリセット・都道府県・サービス種別・並び順・担当者・対応ツール）はアコーディオンに収納します。
- **管理モードのインライン編集** — ステータスバッジ・担当者バッジ・案件回数・今年度利用数を、その場のポップオーバーで編集できます（オーバーレイ外クリックや Escape で閉じる）。
- **添付ファイル** — ファイル選択とドラッグ＆ドロップに対応。1回の報告につき最大5件。
- **日付の扱いに関する注意** — `toISOString()` は UTC に変換されるため日本時間と最大9時間ずれます。日付は必ず `getFullYear()/getMonth()/getDate()` でローカル時間として整形してください。

## 7. 構築手順（ゼロから動かすまで）

ここでは、何もない状態からシステムを動かすまでの手順を順に示します。外側（Google 基盤）から内側（コード）へ、最後にデプロイと検証、という流れです。各段階は前の段階の完了を前提とします。

### 段階0：前提を整える
- **Google Workspace ドメイン**を用意する（Web App のドメイン制限はこのドメイン内に効く）。
- スプレッドシートと GAS の**所有者となる管理用 Google アカウント**を用意する。
- 開発ツールを入れる：Node.js 18 以上、npm、`@google/clasp`（`npm i -g @google/clasp`）、git。
- Zoom 連携を使う場合は、Zoom の Server-to-Server OAuth アプリの認証情報を用意する（後で設定シートへ。**コードには書かない**）。

### 段階1：Google 基盤
1. **申込フォーム**を作成する（氏名・メール・事業所名・相談内容・都道府県・サービス種別）。回答は回答スプレッドシート `<FORM_RESPONSES_ID>` に保存される。
2. **アプリ用スプレッドシート** `<SPREADSHEET_ID>` を新規作成する。
3. アプリ用スプレッドシートに「案件リスト」シートを作り、A1 に IMPORTRANGE を設定して、回答から必要列だけを左詰めで取り込む：
   ```
   ={IMPORTRANGE("<FORM_RESPONSES_ID>","フォームの回答 1!A:E"),
     IMPORTRANGE("<FORM_RESPONSES_ID>","フォームの回答 1!H:I")}
   ```
   初回はセルの「アクセスを許可」を一度押す。
4. 添付保存用の**ドライブフォルダ** `<ATTACHMENT_FOLDER_ID>` を作成する。
5. 必要なら**共有カレンダー／チームカレンダー**を作成する。

### 段階2：データモデル（シート作成）
起動に最低限必要なのは次の3シート。残りは段階5の初期化と自動移行で生成されます。
- 「案件リスト」（段階1で作成済み・書き込み禁止）
- 「サポート記録」（19列、第5章の定義どおり）
- 「タダメンマスタ」（管理者を1名登録：B=氏名 / C=メール / D=admin / E=true）

> 初回アクセス時に `ensureAttachmentSchema_()` が走り、不足列や新シート（年間利用補正・メール下書き等）をスキーマ版6まで自動追補します。全列を手作業で作る必要はありません。

### 段階3：GAS プロジェクト初期化
```
clasp login
# リポジトリ直下に .clasp.json を作る（.gitignore 対象・手動作成）
```
`.clasp.json` の中身：
```json
{ "scriptId": "<SCRIPT_ID>", "rootDir": "." }
```
GAS エディタ側で：
1. 「サービス」＋ から **Gmail API v1**（userSymbol を `Gmail`）と **Google Calendar API v3**（`Calendar`）を追加する。
2. `コード.js` の `SPREADSHEET_ID` を `<SPREADSHEET_ID>` に書き換える。
3. `appsscript.json` を次の内容にする（タイムゾーン・サービス・スコープ・Web App 設定）：
```json
{
  "timeZone": "Etc/GMT-9",
  "dependencies": { "enabledAdvancedServices": [
    { "userSymbol": "Gmail",    "version": "v1", "serviceId": "gmail" },
    { "userSymbol": "Calendar", "version": "v3", "serviceId": "calendar" }
  ]},
  "oauthScopes": [
    "https://mail.google.com/",
    "https://www.googleapis.com/auth/calendar",
    "https://www.googleapis.com/auth/drive",
    "https://www.googleapis.com/auth/spreadsheets",
    "https://www.googleapis.com/auth/userinfo.email",
    "https://www.googleapis.com/auth/script.external_request",
    "https://www.googleapis.com/auth/script.scriptapp",
    "openid"
  ],
  "runtimeVersion": "V8",
  "webapp": { "executeAs": "USER_ACCESSING", "access": "DOMAIN" }
}
```

### 段階4：コード配置とプッシュ
- 配置するファイルは `index.html` / `コード.js` / `appsscript.json` の3つのみ。`src/` は作らない、`temp/` は配布しない。
- `.claspignore` に `node_modules/` 等が含まれることを確認してから：
  ```
  clasp push --force
  ```
  `--force` を付けないと、まれに変更がサイレントにスキップされます。

### 段階5：初期セットアップ関数の実行
GAS エディタの実行ボタンから順に実行します。
```
setupSettingsSheet()          // 「設定」シートを作る
addEmailTemplates()           // メールテンプレート
addForcedCcSetting()          // 自動CC
addMailDryRunSetting()        // ドライラン設定（最初は true 推奨）
addUsageLimitSettings()       // 年間10 / 案件3 の上限
addAttachmentFolderSetting()  // 添付フォルダ
```
そのあと設定シートで `ATTACHMENT_FOLDER_ID`・カレンダーID・（必要なら）Zoom 認証情報を入力します。**動作確認が済むまで `MAIL_DRY_RUN=true`**（外部に送信されません）。

### 段階6：ローカル開発環境
GAS なしで画面を確認できます。
```
git clone <repo> && cd tadasaposys
npm install
npx serve -s . -l 3000   # → http://localhost:3000
# 代替: python -m http.server 3000 --directory .
```
ローカル判定は `typeof google === 'undefined'`。内蔵モックデータ（14パターン）で全画面を確認できます。

### 段階7：テスト
```
npm run test:unit   # Jest 単体（純粋関数）
npm test            # Playwright E2E
```

### 段階8：デプロイ
詳細は第8章。初回のみ新規デプロイで URL と deploymentId を発番し、以降はその**固定 deploymentId へバージョンアップ**します。

### 段階9：受け入れ検証
第11章の受け入れ基準を実施し、問題なければ `MAIL_DRY_RUN=false` にして実送信を有効化します。本番固有 ID は安全な運用記録に控えます。

## 8. デプロイと運用

### 8.1 デプロイの大原則：URL を変えない

**運用開始後に新規デプロイを作ってはいけません。** 新規デプロイは Web App の URL を変えてしまい、タダメンへ案内済みのブックマーク・QR・メールリンクがすべて失効します。必ず**最初に発番した固定 deploymentId へ、`-i` を付けてバージョンアップ**します。

### 8.2 デプロイ手順

```bash
# 0. appsscript.json の webapp が USER_ACCESSING / DOMAIN か確認
# 1. 先に git commit（clasp pull でローカルが上書きされる事故を防ぐ）
git add index.html コード.js appsscript.json
git commit -m "feat: vX.X.X - 説明"

# 2. GAS にプッシュ（--force 必須）
clasp push --force

# 3. 固定 deploymentId へバージョンアップ（初回のみ -i なしで新規発番）
clasp deploy -i <DEPLOYMENT_ID> -d "vX.X.X"

# 4. GAS デプロイ管理画面で「実行=アクセスしているユーザー / アクセス=ドメイン内全員」を目視確認
```

### 8.3 ロールバック

GAS のデプロイはバージョン管理されています。GAS エディタの「デプロイを管理」から、対象デプロイの「バージョンを編集」で過去バージョンに戻せます。git で戻す場合は、過去コミットの `index.html` / `コード.js` をチェックアウトして `clasp push --force` → 固定 deploymentId でデプロイし、その後ローカルを元に戻します。

### 8.4 よくある障害と対処

| 症状 | 確認・対処 |
|------|-----------|
| アプリが起動しない（白画面） | GAS で `doGet` / `getInitialData()` を直接実行しエラーを確認。`SPREADSHEET_ID` の誤り、設定シート未作成（→`setupSettingsSheet()`）、Advanced Service 未有効化が典型 |
| ログインできない | アクセス中のメールがタダメンマスタ C列にあるか、D列（ROLE）と E列（IS_ACTIVE=true）が正しいか確認 |
| メールが送られない | `MAIL_DRY_RUN` が true でないか、Gmail Advanced Service が有効か、実行ログのエラーを確認 |
| カレンダー連携が動かない | Calendar Advanced Service の有効化、`SHARED_CALENDAR_ID`、対象カレンダーの編集権限を確認 |

## 9. セキュリティ

本システムは**介護事業所の個人情報**（氏名・メール・相談内容など）を扱います。そのため、セキュリティは設計・実装・運用の全段階で最優先事項です。本章はセキュリティに関する要件・実装・運用・残課題を独立してまとめます。

### 9.1 セキュリティ設計の原則

基本方針は**多層防御**です。1つの対策に頼らず、認証・認可・入力検証・最小権限・監査の各層を重ねます。特に「画面で隠す」だけのUI制御は防御とみなさず、**必ずサーバー側（バックエンド）でも判定**します。セキュアコーディングは次の5つの視点で実装・レビューします。

1. 認証・認可
2. 入力検証 / 出力無害化
3. 機密情報の管理
4. データ保護 / 最小権限
5. 監査性 / エラー処理

### 9.2 認証（誰であるかの確認）

- **ドメイン制限**：Web App は `access: DOMAIN` に固定し、運営の所属ドメイン外からはそもそもアクセスできません。
- **本人特定**：`executeAs: USER_ACCESSING`（アクセスしている本人として実行）に固定することで、`Session.getActiveUser()` が**確実にログイン本人のメール**を返します。これがなりすまし防止と監査の土台です。
- **スタッフ照合**：実行ユーザーのメールがタダメンマスタに存在し、かつ有効（IS_ACTIVE=true）でなければ、**全処理をブロック**してエラーを返します。未登録・無効ユーザーは何も操作できません。

### 9.3 認可（何をしてよいかの制御）

- **役割ベース**：タダメンマスタの ROLE（admin / staff）で権限を分けます。判定は後方互換として設定の `ADMIN_EMAILS` も参照しますが、ROLE を優先します。
- **サーバー側で強制**：管理者専用の操作は、関数の冒頭で管理者権限を確認する `requireAdmin_()` を通します。画面でボタンを隠すかどうかに関わらず、**権限のない呼び出しはバックエンドで拒否**されます。
- **案件単位の編集権限**：案件の更新系は `ensureCaseEditableByActor_()` を通し、**担当者・サブ担当・管理者以外の更新を拒否**します。
- **モードと操作範囲**：通常モードは自分の担当のみ操作可（未対応は共有）、閲覧モードは読み取り専用、管理モードは管理者のみ全操作可。
- **年間・案件回数のガード**：年間利用が上限以上の案件はアサイン不可（「回数超過」導線のみ）。案件回数が上限に達した案件は再開不可。これらもサーバー側ロジックで担保します。

### 9.4 入力検証 / 出力無害化

- **数式インジェクション対策**：スプレッドシートへ書き込む値は `sanitizeForSheet_()` で無害化します。`=`、`+`、`-`、`@` などで始まる文字列はスプレッドシート上で数式として解釈され得るため、これを防ぎます。外部入力・シート値・GAS で書き込む値はすべて検証・無害化の対象です。
- **設定更新のホワイトリスト**：`updateSettingsAdmin()` は**あらかじめ許可したキーのみ**を更新し、それ以外のキーへの書き込みは拒否します。これにより設定経由の不正な値注入を防ぎます。
- **出力（XSS）への配慮**：React は既定で値をエスケープして描画します。`dangerouslySetInnerHTML` のような生 HTML 挿入は使わない方針とし、ユーザー由来の文字列をそのまま HTML として注入しないようにします。

### 9.5 機密情報の管理

- **コード・ドキュメント・GitHub に置かない**：Zoom の OAuth 認証情報などの機密は、**操作者の許可があってもコードやドキュメント、GitHub に保存しません**。設定シートや PropertiesService など、アクセス制御された場所で管理します。
- **ハードコーディング原則禁止**：固定値は事前に「設定化できないか」を検討します。
- **環境固有 ID の扱い**：スプレッドシート ID・スクリプト ID・デプロイ ID は環境ごとの鍵に準じる情報として、本書のような共有ドキュメントには直書きせず、安全な運用記録で管理します（本書冒頭の方針）。
- **共有ストレージへの配慮**：本書を含むドキュメントを Google ドライブ等で共有する場合は、**共有範囲を組織内に限定**します。リンク公開（全員／リンクを知る全員）にしてはいけません。

### 9.6 データ保護 / 最小権限

- **アクセス境界**：Web App のドメイン限定により、データに触れられるのは所属ドメイン内の認証済みユーザーだけです。
- **案件リストの保護**：案件リストは書き込み禁止とし、IMPORTRANGE 由来の一次データを壊しません。補正は分離した「案件補正」シートで行います。
- **OAuth スコープ**：付与するスコープは機能に必要な範囲（メール・カレンダー・ドライブ・スプレッドシート・本人メール取得・外部リクエスト・スクリプト管理）に限定します。

### 9.7 監査性 / エラー処理

- **監査ログ**：管理者の操作（再アサイン・ステータス変更・案件編集・設定変更など）は、**操作者・対象・変更前後の値（BEFORE/AFTER の JSON）とともに監査ログシートへ記録**します。誰が何をいつ変えたかを後から追跡できます。
- **失敗時は安全側へ**：処理が失敗した場合は中途半端に進めず、安全に停止してエラーを返します。
- **ドライランによる事故防止**：メール送信を伴う検証は `MAIL_DRY_RUN=true` で外部送信せずに行えます。テストで作成・更新したデータは、テスト後に必ず削除または復元し、元の状態へ戻します（戻せない検証は実施しない）。

### 9.8 デプロイ・運用上のセキュリティ

Web App の設定は**必ず `executeAs: USER_ACCESSING` / `access: DOMAIN`** に固定します。次の設定は**禁止**です。

- 実行ユーザー＝自分（`USER_DEPLOYING`）→ 監査・権限境界が崩れる。
- アクセス＝Google アカウントを持つ全員（`ANYONE_WITH_GOOGLE_ACCOUNT`）／全員（`ANYONE`）→ ドメイン外へ情報が流出する。

**過去に「実行＝自分／アクセス＝Google アカウント全員」という誤設定で運用された事故**があります。介護事業所の個人情報を扱うため、この再発は固く禁じます。デプロイ前に `appsscript.json` の webapp 設定を差分で確認し、デプロイ後は GAS のデプロイ管理画面で目視確認してください。

### 9.9 既知のセキュリティ上の留意点

コードを確認した範囲では、案件の更新系・管理操作には担当者／管理者のチェックが一貫して入っています（`ensureCaseEditableByActor_()` または `requireAdmin_()`）。そのうえで、運用上とくに意識すべき点を挙げます。

- **スプレッドシート自体のアクセス権が最終的な境界**：データも設定（Zoom 認証情報を含む）もすべてスプレッドシートに保存されます。Web App はアプリ経由のアクセスを制御しますが、スプレッドシートを直接共有された人はアプリを介さず中身を読めます。**スプレッドシートの共有範囲を必要最小限に限定**してください。これが実質的な最終防衛線です。
- **管理機能による不整合データ（v1.11.5 以前）**：統一ゲートキーパー導入前に作られた不整合データが残っている可能性があります。現在の操作は整合性が保証されますが、古いデータは点検・修復の検討対象です。
- **新着バッジの既読状態は localStorage 依存**：セキュリティ影響はありませんが、ブラウザをまたいで同期されません。

## 10. テストと品質保証

品質は2層のテストで担保します。

- **単体テスト（Jest）**：UTC ずれを起こさない年度計算、数式インジェクション対策、メール正規化、真偽値パースなどの**純粋関数**を検証します。実行は `npm run test:unit`。
- **E2E テスト（Playwright）**：アプリ初期化、案件一覧、表示モード切替、検索・フィルタ、案件詳細、管理者機能、アクセシビリティ（axe-core による WCAG 2.1 AA 確認）を、実ブラウザで検証します。実行は `npm test`。

依存は `package.json` の devDependencies（`@playwright/test`、`@axe-core/playwright`、`jest`、`typescript` 等）で管理します。CI は GitHub Actions のワークフローで実行できます。

## 11. 検証・受け入れ基準

構築完了とみなすには、次のすべてを満たす必要があります（各項目はテスト可能であること）。

1. Web App の URL にアクセスし、タダメンマスタ登録アカウントでログインできる。
2. フォーム申込が案件リスト経由で「未対応」案件として表示される。
3. アサイン → 日程確定（Meet 発行）→ 記録 → 完了報告の一連が完了する。
4. 年間10回 / 案件3回の上限ガードが作動する（超過時に「回数超過」導線になる）。
5. 管理モードでステータス変更・再アサイン・上限特例が反映され、監査ログに記録される。
6. `npm run test:unit` と `npm test` がともに成功する。
7. `appsscript.json` の webapp が `USER_ACCESSING` / `DOMAIN`（差分と GAS 管理画面の両方で確認）。
8. デプロイが固定 deploymentId へのバージョンアップで行われ、URL が不変である。
9. ローカル（`npx serve`）でモックデータの全画面が表示される。
10. 検証完了後に `MAIL_DRY_RUN=false` へ切り替え、実送信が行われる。
